# 退出复盘

循环正常退出(目标状态全部达成)时产出一次复盘,随退出汇报交出。产物是**候选**而非结论:裁决与落地不在本文件范围内,由后续的人工回应异步触发,本文件只产出候选并浮现。

异常退出(收敛兜底中的「暂停询问用户」各条)不产复盘——那是升级,不是干净收敛。

## 基调:空即成功

复盘价值的前提是这一条:**绝大多数干净收敛的 PR,下列四类候选均为空。空是预期结果、是循环顺利收尾的标志,不是待填补的缺漏。**

返回「无候选」是正常结果。不要为了让复盘显得有产出,把已实施的常规修复、reviewer 的例行建议、本就在 issue 范围内的改动强行归入 ADR / CONTEXT / CLAUDE.md / follow-up。虚构原则升级、记录显而易见的概念、为范围内的工作立项,这些噪音会淹没少数真正的信号,比留空更糟。一条候选只有在能用一句话说清「它为何超出已有记录或已有范围」时才值得写。

## 五段结构

### 1. 过程总结(简短陈述,几行即可)

客观快照,不评判:

- 轮数(`round_estimate` 量级)
- 参审 reviewer(哪几家过审、哪家因故障停用)
- 关键修复(真正改变行为的一两处,非 nit)
- 在案 pushback(驳回内容与依据,各一行)
- 故障(如有:哪家 bot 出过何种问题、如何裁决)

干净的 PR 此段通常五六行。

### 2. ADR 候选

仅当循环中**浮现一条 `docs/adr/` 尚未记录的真原则**时写——通常是被 reviewer 跨多个 HEAD 反复提出、最终以一个具普遍适用性的取舍敲定的决策(会在后续场景反复套用的口径)。一次性的局部修复不是原则,`docs/adr/` 已有的不重复。无则写「无」。

### 3. CONTEXT.md 更新候选

仅当某个概念的**当前形状确已改变**、而 CONTEXT.md 仍停在旧形状时写(CONTEXT.md 只记此刻是什么,不记 roadmap 与过程)。新增常规实现、修复 bug 都不改变概念形状,不在此列。无则写「无」。

### 4. CLAUDE.md 更新候选

仅当循环中浮现一条**能预防同类返工的 agent 工作规范**、且仓库 CLAUDE.md（含其引用的规范文档）尚未涵盖时写——典型信号:同类 reviewer 意见反复出现而根因是仓库约定未写明、质量门或工具链的隐性坑使修复多走弯路。一次性的修复经验不是规范,CLAUDE.md 已有的不重复。无则写「无」。

### 5. follow-up issue 候选

仅当循环中遇到**确实超出本 issue 范围**的工作时写——一项按范围收口、但应单独立项的缺陷或改进。范围内的收尾、已实施的修复都不算。立项是用户的决定,此处只提候选,不代为开 issue。无则写「无」。

## 边界

产出候选 → 浮现 → 止:候选批准后由谁、以何种方式落地(写 ADR / 改 CONTEXT.md / 补 CLAUDE.md / `/to-issues` 立项),本文件不指定。
